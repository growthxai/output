---
title: "LLM Costs"
description: "Observe per-call LLM token usage and computed cost via the llm:call_cost event"
---

The `@outputai/llm` package emits a `llm:call_cost` event after each `generateText` and `streamText` call. Use it to log costs, send metrics, or aggregate spend per workflow or activity. The event is delivered through the same [hooks system](/operations/error-hooks) as error hooks: register with `on('llm:call_cost', handler)` from `@outputai/core/hooks` in a file listed under `output.hookFiles` in `package.json`.

## Setup

1. Create a hook file and import `on` from `@outputai/core/hooks`.
2. Register the handler for `llm:call_cost`.
3. Add the file path to `output.hookFiles` in `package.json`.

See [Error Hooks â€” Setup](/operations/error-hooks#setup) for the same pattern (hook file + `output.hookFiles`). Use `on('llm:call_cost', ...)` instead of `onError(...)`.

Example:

```javascript
// src/llm_cost_hooks.js
import { on } from '@outputai/core/hooks';

on('llm:call_cost', async ({ workflowId, activityId, modelId, usage, cost }) => {
  console.log('LLM call', { workflowId, activityId, modelId, usage: usage?.totalTokens, cost: cost?.total });
});
```

## Payload

The handler receives a single object with the following fields.

| Field | Type | Description |
|-------|------|--------------|
| `activityId` | `string \| undefined` | Temporal activity ID of the step/evaluator that made the LLM call. Present when the call runs inside a workflow activity. |
| `workflowId` | `string \| undefined` | Temporal workflow execution ID. Present when the call runs inside a workflow. |
| `modelId` | `string` | Model identifier (e.g. `claude-sonnet-4-20250514`, `openai/gpt-4o`). From the prompt file config. |
| `usage` | `object` | Token usage for this call. Same shape as the AI SDKâ€™s [totalUsage](https://ai-sdk.dev/docs/reference/ai-sdk-core/generate-text#total-usage) (aggregated across all steps for that call). |
| `cost` | `object` | Computed cost for this call. See [Cost structure](#cost-structure) below. |

### Usage shape

`usage` is the `totalUsage` from the AI SDK response. It aggregates input, output, cached input, and optional reasoning tokens. See the [AI SDK generateText â€” totalUsage](https://ai-sdk.dev/docs/reference/ai-sdk-core/generate-text#total-usage) reference for the full type.

Example:

```json
{
  "inputTokens": 217,
  "inputTokenDetails": { "noCacheTokens": 217, "cacheReadTokens": 0, "cacheWriteTokens": 0 },
  "outputTokens": 9,
  "outputTokenDetails": { "textTokens": null, "reasoningTokens": null },
  "totalTokens": 226,
  "reasoningTokens": null,
  "cachedInputTokens": 0
}
```

When the model uses extended thinking (e.g. reasoning tokens), `reasoningTokens` and `outputTokenDetails.reasoningTokens` may be set. Cached prompt tokens are reported in `cachedInputTokens` and in `inputTokenDetails.cacheReadTokens`.

### Cost structure

`cost` is produced by the LLM packageâ€™s cost calculator from `usage` and the modelâ€™s per-million-token pricing. Pricing is fetched from a built-in source and cached; see the [Cost Estimation](/operations/cost-estimation) guide for how pricing is defined and overridden.

| Field | Type | Description |
|-------|------|-------------|
| `total` | `number \| null` | Total estimated cost in dollars for this call. `null` if pricing is missing, fetch failed, or the model is unknown. |
| `message` | `string` | Present when `total` is `null`; describes why cost could not be computed (e.g. `"Missing cost reference for model"`, `"Failed to fetch models pricing"`). |
| `components` | `object` | Present when the cost was calculated successfully. Per-token-type breakdown; each component is either `{ value: number }` or `{ value: null, message: string }`. |

**Components:**

| Component | Description |
|-----------|--------------|
| `input` | Cost for non-cached input (prompt) tokens. |
| `cachedInput` | Cost for cached input tokens (e.g. prompt cache read). |
| `output` | Cost for output (completion) tokens. |
| `reasoning` | Cost for reasoning/thinking tokens, when the provider exposes them and pricing includes a reasoning rate. Omitted when the provider does not separate reasoning from output. |

Example when cost was calculated successfully:

```json
{
  "total": 0.002341,
  "components": {
    "input": { "value": 0.001302 },
    "cachedInput": { "value": 0 },
    "output": { "value": 0.001039 },
    "reasoning": { "value": 0.00027 }
  }
}
```

Example when a component has no price (that component has `value: null` and `message`; `total` sums only the components with numeric values):

```json
{
  "total": 0.002611,
  "components": {
    "input": { "value": 0.001302 },
    "cachedInput": { "value": null, "message": "Missing cache input cost" },
    "output": { "value": 0.001039 },
    "reasoning": { "value": 0.00027 }
  }
}
```

Example when cost could not be calculated (unknown model or pricing unavailable):

```json
{
  "total": null,
  "message": "Missing cost reference for model"
}
```

Cost is computed as `(tokens / 1_000_000) * pricePerMillion` for each component. Non-cached input uses `inputTokens - (cachedInputTokens ?? 0)`. If any required component has no price, that component has `value: null` and a `message`; `total` still sums the components that have a numeric `value`.

Handler errors are caught and logged by the framework; they do not affect the workflow or the LLM call.
